# Deephome 意图搜索引擎 —— 设计文档

> 一句话：把「用户说出来的词」还原成「他真正的意图」，再把意图翻译成「可查询的房源属性」，并让整个过程可解释、可纠错。

---

## 1. 从第一性原理定义问题

一次找房搜索里，其实有三层东西，传统搜索把它们混为一谈：

| 层 | 例子 | 谁拥有它 |
|---|---|---|
| **表层词 (surface)** | "大窗户" | 用户嘴上说的 |
| **真实意图 (intent)** | "采光好 / 房子明亮" | 用户心里想的 |
| **可观测属性 (attribute)** | 朝南、高楼层、无遮挡、开窗比例大、层高 | 房源数据里能查的 |

核心洞察：**用户的词，是意图的「代理」，不是意图本身。**
"大窗户" 只是用户能想到的、用来描述 "我要一个明亮房子" 的一个近似说法。而真正决定明不明亮的，是朝向、楼层、遮挡——这些用户**根本没提，甚至不知道该提**。

- 传统搜索（含 Repliers `/nlp`）匹配的是**代理词**：有没有 "big windows" 这个字符串。
- Deephome 要匹配的是**意图**：这套房子到底采光好不好。

这就是整个产品的立命之本。下面所有设计都是为这一句话服务的。

---

## 2. 为什么现有方案都不够（市场对标 + 反推）

调研了当前市面上的做法，逐一反推为什么它们单独用都不对：

**① Repliers `/nlp`（字面参数抽取）**
它把 "enclosed balcony" 规范化成 `balcony=encl`，把 "Annex" 抽成 `neighborhood=Annex`。遇到 "大窗户"，它要么丢掉，要么塞进关键词/图搜。它**不会**、也**不该**推出 "朝向"——因为它的定位是「自然语言 → MLS 字段翻译器」，不是「意图理解器」。这正是 Deephome 的差异化空间：我们坐在它上面一层。

**② 纯 LLM 每次自由展开意图**
让大模型每次现场把 "采光好" 展开成属性。问题：不可审计、不稳定、会幻觉映射、且按 token 计费。产品的核心资产不该是「一段 prompt」。

**③ 纯向量 embedding 语义检索**
把 query 和房源都 embed，算相似度。问题：它把 surface 和 intent 揉在一个向量里，**不可解释、不可纠错**——而「解释为什么」恰恰是本产品的灵魂。

**④ 纯确定性解析器**（规则 + 同义词，无 LLM）
稳定、便宜，但对**新说法、歧义、否定、上下文**很脆弱。

**⑤ 业界最前沿的 agentic NLQ**（LLM 路由 + 向量库 + cross-encoder rerank + 硬过滤）
方向对，但偏「通用检索工程」，没把「房产意图」沉淀成一个**可拥有、可改进的领域资产**。

**结论**：任何单一路线都不对。最优解是把各家长处组合，核心是引入一个显式的领域资产——**意图知识库**。

## 3. 最优方案总览

```
用户 query
   │
   ▼
① 意图理解 (LLM 前端，可离线回退)  ──grounded by──►  [意图知识库 KB]  ◄── 产品核心资产
   │  输出：意图帧 IntentFrame                              （意图 → 属性 的映射，可审计/可改进）
   ▼
② 意图 → 属性 规划 (查 KB 展开)
   │  输出：查询计划 QueryPlan（硬过滤 + 加权软信号 + 文本词 + 图搜概念 + 地点信号）
   ▼
③ 混合检索 (数据源可插拔：Repliers / 本地样例)
   │  结构化过滤召回 + 多信号打分
   ▼
④ 透明排序 + 解释生成
   │  线性可解释融合；cross-encoder 仅作 top-K 可选精修
   ▼
⑤ 反馈闭环：用户纠正意图 → 重排 → 沉淀为对 KB 的修正
```

**一句话概括最优性**：
> LLM 做它擅长的模糊活（把新说法对齐到已知概念、消歧、否定、上下文）；
> KB 做必须正确的活（意图 → 加权属性）；
> 混合检索做召回；透明线性模型做可解释排序。
> —— 既拿到 LLM 的语言灵活性，又不牺牲可审计 / 可解释 / 可改进 / 低成本。

## 4. 架构分层与每层的「为什么」

**Layer 0 · 富化（离线/入库时）** —— 已实现 (`src/enrich.ts`)
把 MLS 原始房源规范化，并**派生用户真正在意、但 MLS 没有直接字段的属性**：朝向、楼层、层高、可养宠、以及**开窗比例**。每个派生值都带**来源 + 置信度**（provenance）。其中开窗比例可由 `VisionAdapter` 从**照片**派生，并以更高置信度覆盖措辞猜测（见 `src/vision.ts`）。
*为什么*：这是让 "采光好" 变得可查询的地方。没有这一层，意图无处落地。

**Layer 1 · 意图理解** → `IntentFrame`
把 query 切成原子需求，每个判定为：字面约束 / 已知意图概念 / 未知需求；并判断 极性(想要/避免)、软硬(必须/偏好)、强调权重、置信度；歧义时产出**至多一个**澄清问题。
*为什么至多一个*：过度追问会毁掉体验。只在意图**实质分叉**时问（"大窗户" = 采光 vs 景观 vs 空间感）。

**Layer 2 · 意图 → 属性 规划** → `QueryPlan`
对每个意图概念查 KB，展开成：硬过滤（仅当用户明确"必须"）、加权软信号（带打分函数）、文本词、图搜概念、地点信号。
*为什么*：这一层就是「翻译」。用户从没提过"朝向"，但因为他要"采光好"，我们把朝向加进了排序依据——这一步就是 Deephome 的全部价值所在。

**Layer 3 · 混合检索**（数据源可插拔）
先用硬约束缩小候选，再对候选做多信号打分。

**Layer 4 · 透明排序 + 解释**
线性加权融合四个分支，权重来自计划在各分支的投入占比。每套房生成「为什么」——直接来自命中的属性，不是模型编的。

**Layer 5 · 反馈闭环**
用户纠正意图即时重排；长期把纠正沉淀为对 KB 映射的修正。**KB 是会随使用变聪明的学习资产**。

## 5. 为什么检索必须是「混合」的

以 "采光好" 为例，证据天然分散在四个地方，任何单支都不完整：

| 分支 | 采光的证据 |
|---|---|
| 结构化 | 朝向 = 南、楼层 = 高、层高 |
| 文本 (remarks) | "sun-filled", "south-facing", "unobstructed" |
| 图片 | 照片里的大窗、明亮客厅 |
| 地点 | （对"安静""通勤"等意图）周边主干道/地铁 |

朝向在结构化字段里、"阳光充足" 在文本里、大窗在照片里——**没有任何一个单独够用**。所以混合是必需，不是装饰。

---

## 6. 为什么排序坚持「透明线性」而非黑盒模型

Deephome 的灵魂是「解释为什么」。每一分都必须能拆回到「哪条意图 / 哪个属性」。
因此最终融合是一个透明线性模型（各分支归一化后加权求和，权重 = 计划在该分支的投入占比）。
真正的 cross-encoder / instruction-following reranker 可以作为 **top-K 的可选精修**，但**绝不能取代**这套可解释骨架。可解释 > 一点点排序精度。

---

## 7. 与 Repliers 免费档如何结合

- Preview（$0）档提供：NLP Search、Image Search、全文关键词、结构化过滤——但只针对 **sample data**。
- 真实在售/在租房源需 Standard（$199/月）+ MLS 牌照授权。

因此：**意图引擎完全可以用免费档搭建并证明**，因为它的价值在"理解与排序"，不在"房源多少"。数据源做成可插拔接口 `ListingSource`：
- `LocalSampleSource`：离线样例，零 key 跑通全流程。
- `RepliersSource`：真实适配器，把 `QueryPlan` 映射成 `/listings` 参数 + 关键词 + `imageSearchItems`。

照片派生的开窗比例用的是 Repliers **AI Image Search**（免费 Preview 档即可），见 `RepliersVisionAdapter`。

## 9. 演示结论（用数据说话）

同样的表层词 "大窗户"，在不同意图下给出**完全不同**的排序——这正是意图搜索的证明：

| 场景 | 朴素关键词 Top1 | Deephome Top1 | 说明 |
|---|---|---|---|
| "大窗户 **采光好**" | **C002**（朝北·3楼，字面写着 big windows） | **C001**（朝南·28楼） | 字面命中≠采光好；C002 沉出前 5 |
| "大窗户 **景观好**" | C002 / 同上 | **C012**（朝北·33楼） | 景观看楼层，朝北无所谓；同一朝北房，这里反而上位 |

视觉层进一步证明：文案吹 "big windows" 但照片其实普通窗 → 开窗比例从措辞的 0.70 **下修**到照片的 0.40；文案只字未提但照片是落地窗 → 从默认 0.45 **上修**到 0.90。**以照片为准，防文案注水、补文案遗漏。**

## 10. 现状与下一步

已完成：意图引擎（L1–L4）+ 富化管线（L0）+ 照片派生开窗比例（视觉层）+ 可运行离线演示。

下一步，按优先级：

1. **评测 harness（最关键）**：用固定金标 query 集 + LLM-as-judge，量化每次改 KB/权重/视觉到底有没有让排序更准。没有评测就没有"最优"。
2. **批量视觉打分**：`analyzeMany` 目前逐套调用，生产应一次 image-search 给整个候选集打分。
3. **真实语义分支**：现在文本分支是 BM25-lite；生产接 embedding 向量检索。
4. **KB 学习闭环**：把用户对澄清/结果的纠正回流成对映射权重的修正，让 KB 越用越准。
5. **地点/地名真实化**：`/locations`、`/places` 端点替换样例 POI。
6. **交易类型默认值**：买/租不应混排；query 未指明时应默认或追问（当前是已知小限制）。
