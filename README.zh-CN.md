# Deephome

**面向多伦多（及加拿大）的意图式找房。** 传统搜索匹配你输入的**词**；Deephome 还原你真正**想要的东西**，再按能真正满足该意图的属性来排序——并解释每一个结果。

<p align="center"><a href="./README.md">English</a> · <b>中文</b></p>

> 「我要大窗户」的真实意图，往往是**「我要采光好」**——而采光由朝向、楼层、遮挡、开窗比例决定，*这些用户一个都没提*。关键词搜索只会去匹配「big windows」这串字；Deephome 做的是**表层词 → 真实意图 → 可查询属性**的翻译。

这是一个非营利、开源的工具。无广告、不倒卖数据，只是想把找房这件事做得更好。

## 一个例子说明白

查询：**「大窗户 / 采光好」**

| | 朴素关键词搜索 | Deephome（意图） |
|---|---|---|
| 排第一 | 一套**朝北、3 楼**、文案里写着「big windows」的房 | 一套**朝南、28 楼**、整面落地窗的房 |
| 为什么 | 命中了字面短语 | 还原出「采光好」→ 按 朝向 + 楼层 + 开窗比例 排序 |

同样这句话，换成**「大窗户 / 景观好」**，反而是高层朝北的房排第一——因为景观看楼层，不看太阳。**同样的词、不同的意图、相反的排序**。这是任何关键词或纯向量搜索都做不到的。

## 它是怎么工作的

一条分层流水线，核心资产是一个**可审计的意图知识库**（表层词 → 意图 → 加权属性）：

- **第 1 层 · 意图理解。** LLM 可插拔前端（带确定性离线兜底），由知识库 grounding。处理新说法、否定、歧义；只有当意图**实质分叉**时才追问一句。
- **第 2 层 · 意图 → 属性规划。** 把每个意图展开成 硬过滤 + 加权软信号 + 文本/图像概念 + 周边地点信号。
- **第 3 层 · 混合检索。** 结构化 + 全文 + AI 图像 + 地点——因为一个意图的证据分散在这四处，缺一不可。
- **第 4 层 · 透明排序。** 线性、可解释的融合；每一分都能追溯到「哪个意图、哪个属性」。（cross-encoder 重排只是 top-K 的可选精修，绝不替代。）
- **第 0 层 · 富化。** 把原始 MLS（文案 + 基础字段 + 照片）变成意图所需的属性——朝向、楼层、层高、宠物政策，以及**从照片派生的开窗比例**——每个都带来源与置信度。

完整设计与推理见 [`DESIGN.md`](./DESIGN.md)。

## 快速开始（离线，无需任何 API key）

```bash
npm install
npx tsx demo.ts          # 意图搜索——表层词 vs 真实意图
npx tsx demo-enrich.ts   # 原始 MLS → 富化属性（带来源/置信度）
npx tsx demo-vision.ts   # 开窗比例从照片来，而非措辞猜
```

全部基于内置样例数据、零 key 即可运行。若本机没有 `tsx`，用 `npm run build:run`。

## 真实数据（Repliers）

数据源可插拔（`ListingSource`）。加拿大**不存在免费、可再分发的完整 MLS 数据**——它由 CREA 与地方 board 授权。可行路径：

- **Repliers** 免费 *Preview* 档可在样例数据上搭建并验证整个引擎；真实房源需付费档**加上** MLS/brokerage 授权。
- 照片派生的开窗比例用的是 Repliers **AI Image Search**（免费档即可）。切换只需一行：`enrich(raw, { vision: new RepliersVisionAdapter({ apiKey }) })`。

## 现状与路线

可运行的原型（算法 + 可验证的证明）。下一步：评测 harness（LLM-as-judge）量化每次改动、批量照片打分，以及把用户纠错沉淀为知识库改进的学习闭环。
